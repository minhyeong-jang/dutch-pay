# Dutch Pay - 비즈니스 플랜

> 기획자, 사업개발, 대표 3인의 관점을 종합한 사업 계획서

## 1. 포지셔닝

**한 줄 정의:** "토스로 못 하는 복잡한 정산, 1분 만에"

| 토스/카카오페이 | 우리 앱 |
|----------------|---------|
| 1명이 결제, N명이 나누기 | **여러 명이 각각 결제한 복수 건을 교차 정산** |
| 단일 결제건 분할 | 1차/2차/3차 전체를 한 번에 정산 |
| 해당 앱 사용자만 참여 가능 | 플랫폼 독립적 (아무 송금 앱 사용 가능) |
| 일괄 송금 모드 없음 | 대표자 1명이 모아서 분배하는 일괄 송금 지원 |

**타겟:** 20대 후반~30대, 여행/회식/동호회에서 월 2회 이상 정산이 필요한 사람

## 2. 기능 우선순위

### P0 - MVP 필수 (첫 출시에 반드시 포함)

| 기능 | 설명 |
|------|------|
| 참가자 입력 | 이름으로 참가자 등록 |
| 결제내역 입력 | 사용처, 금액, 결제자, 참여자 선택 |
| 자동 정산 계산 | 누가 누구에게 얼마를 보내야 하는지 산출 |
| 정산 결과 조회 | 개인별 요약 + 일괄/각자 송금 탭 |
| 모임(방) 생성/관리 | 복수 모임 관리 |
| **공유 링크 생성** | 정산 결과를 URL로 공유 (비로그인 접근 가능) |
| **카카오톡 공유** | 정산 결과를 카카오톡으로 공유 (바이럴 핵심) |
| **정산 결과 텍스트 복사** | 클립보드 복사 |
| **모바일 퍼스트 UI** | 정산의 90%는 모바일에서 발생 |
| 서버 데이터 저장 | localStorage → DB 전환 |

### P1 - 중요 (MVP 직후)

| 기능 | 설명 |
|------|------|
| 카카오 로그인 | 소셜 로그인 (카카오 우선) |
| 결제 항목 삭제 | 현재 추가만 가능, 삭제 기능 없음 |
| 실시간 동기화 | 같은 모임 참가자가 동시 편집 |
| 푸시 알림 (모바일) | "OO님이 결제내역을 추가했습니다" |
| 모임 히스토리 | 과거 모임 아카이빙 |
| 정산 완료 체크 | 누가 송금했는지 체크 기능 |

### P2 - 나중에

| 기능 | 설명 |
|------|------|
| 영수증 OCR | 카메라로 영수증 촬영 → 자동 입력 |
| 송금 딥링크 | 토스/카카오페이로 바로 송금 연결 |
| 정기 모임 | 매주/매월 반복 모임 템플릿 |
| 다국어/환율 | 해외여행 시 환율 계산 |
| 통계/대시보드 | 월별 지출 통계 |
| 항목별 불균등 분배 | 비율 지정 정산 |
| 엑셀/PDF 내보내기 | 정산 내역 다운로드 |

## 3. 필수 화면 목록

| # | 화면 | 핵심 요소 |
|---|------|----------|
| 1 | **온보딩/랜딩** | 가치 전달 + "바로 시작하기" CTA (로그인 불필요) |

| 5 | **Step3 - 정산 결과** | 총액, 개인별 요약, 일괄/각자 송금 탭, 공유 버튼 |
| 6 | **공유 결과 페이지** | 읽기 전용, "내 이름 찾기" 하이라이트, 앱 유도 CTA |

| 8 | **네비게이션** | 웹: 상단 헤더 / 앱: 하단 탭 바 |

## 6. 법적 고려사항

| 규제 | 현 단계 해당 여부 | 비고 |
|------|:---:|------|
| 전자금융거래법 | X | 정산 "계산"만 하면 불필요. 직접 송금 중개 시 필요 |
| 개인정보보호법 (PIPA) | O | 서버 저장 시 수집/이용 동의 필수 |
| 정보통신망법 | O | 이용약관 게시, 푸시 알림 수신 동의 |
| 유사수신행위 규제 | X | 자체 포인트/적립금 발행하지 않으면 해당 없음 |
| 마이데이터 | X | 카드사/은행 데이터 연동 시 필요 (자본금 5억+, 초기 비현실적) |

**면책 필수:** "계산 결과는 참고용이며, 금전적 분쟁에 대해 서비스 제공자는 책임지지 않습니다"

## 7. 성장 전략

### 핵심 성장 엔진: 바이럴 루프

```
정산 결과 공유 → 카톡으로 링크 전달 → 받은 사람이 결과 확인
                                      → "나도 써볼까?" → 앱 사용
                                      → 자기 모임에서 정산 → 공유 → 반복
```

### 채널별 전략

| 채널 | 전략 | 시기 |
|------|------|------|
| SEO | "더치페이 계산기", "여행 정산" 키워드 최적화 | 출시 전부터 |
| 카카오톡 공유 | 정산 결과에 앱 링크 자동 포함 | 출시 즉시 |
| 에브리타임 | 대학생 MT/여행 시즌 콘텐츠 | 5월, 9월 |
| 블라인드 | 직장인 회식 정산 관련 글 | 연중 |
| ASO | 앱스토어 키워드 최적화 | 앱 출시 시 |

### KPI

| 단계 | 지표 | 목표 |
|------|------|------|
| 1~3개월 | MAU | 5,000 |
| 1~3개월 | 모임 생성 전환율 | 방문자의 30%+ |
| 1~3개월 | 공유 링크 발생률 | 모임의 40%+ |
| 4~6개월 | WAU | 2,000 |
| 4~6개월 | 바이럴 계수 (K-factor) | 0.5+ |
| 7~12개월 | M1 Retention | 15% |
| North Star | 월간 완료 정산 수 | - |

### 계절성 참고

- **피크 시즌:** 5월(MT), 7~8월(여름 여행), 12월(연말 모임)
- **오프 시즌:** 1~2월, 6월
- 피크 시즌 2주 전에 콘텐츠 마케팅 집중

## 8. 리스크 및 대응

| 리스크 | 심각도 | 대응 |
|--------|:------:|------|
| "빅테크 정산으로 충분하다"는 인식 | 높음 | 앱 설치 없이 웹으로 즉시 사용. "토스는 1건씩, 우리는 통째로" 차별 메시지 |
| 만들다가 지쳐서 포기 | 매우 높음 | MVP 스코프 철저 관리. 80%면 출시. 기술 부채는 나중에 |
| 토스가 같은 기능을 만듦 | 중간 | 속도와 니치에서 이김. 사용자 커뮤니티와 SEO 자산 선점 |
| 로그인 강제 → 이탈 | 중간 | 비로그인 사용 필수 보장. 로그인은 "데이터 영구 저장" 시점에 유도 |
| 정산 결과 정확성 이슈 | 높음 | 정산 알고리즘 단위 테스트 필수. 이용약관에 면책 명시 |
| 모바일 UX 부족 | 중간 | 모바일 퍼스트로 설계. 테이블 → 카드형 UI 전환 |

## 9. 지금 결정해야 할 것 vs 나중에 결정할 것

### 지금 결정

1. 커스텀 도메인 (SEO 직접 영향)
2. 모바일 퍼스트 UI 설계 방향
3. 공유 기능의 기술 방식
4. 출시 목표일 (여름 시즌 전 5월 권장)

### 나중에 결정

1. 네이티브 앱 출시 시점 (웹 성과 보고)
2. 로그인/회원가입 도입 시점 (MAU 5,000 이후)
3. 프리미엄 기능 세부 사항
4. 제휴 파트너 선정
5. UI 라이브러리 마이그레이션 범위
